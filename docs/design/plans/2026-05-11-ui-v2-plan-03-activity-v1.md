## Decisions (locked — no ambiguity for the implementer)

1. **Interestingness package:** `internal/interestingness/` owns the interface definition and one Go file per detector category: `performance.go`, `anomaly.go`, `causation.go`, `failure.go`, `security.go`, `configuration.go`, `novelty.go`. Each detector implements `Detector` (see Task 3.3). No detector logic is allowed outside this package.

2. **`InterestingnessTagged` event kind:** Detectors emit a new event kind (`interestingness.tagged`) into the event store via `eventstore.Appender`. The payload carries `target_event_id`, `category`, `name`, and `explanation`. The Activity service joins on read — it does not denormalise tags into source event rows.

3. **Stories coalescer:** `internal/activity/stories/` owns the coalescer pipeline. It reads a window of events by `correlation_id`, groups them into a `Story` proto message, and writes derived story metadata to a lightweight projection table (not a separate event stream). The coalescer re-runs if coalescing logic changes (pure read from the event store).

4. **Stories tab design:** The v2 design from `screenshots/06-activity-stories-v2-01.png` is canonical. The feed column is `max-width: 720px`; the context rail is `width: 380px`, always visible on screens wider than 1100px. Click a story card → rail populates. No inline expansion.

5. **All Events tab:** Faceted explorer. Facets: Kind, Source, Entity, Issued by — with per-facet counts. A chip-based query bar (`kind:cmd since:1h`) with free-text input. A stacked sparkline color-encoded by event kind at the top. A queryable table beneath. Slide-in detail panel on row click (OTel identity, causation chain, payload JSON, actions).

6. **Saved tab:** Lists saved queries. Each entry shows: name, filter string, last-run time, next-run time (if cron-scheduled), a "Run now" button, and a delete affordance. A "Save current query" button above the list opens a save dialog with name + optional cron expression.

7. **`ActivityService` ConnectRPC:** New service defined in `proto/switchyard/activity/v1/activity.proto`. Methods: `Stories(filter, cursor) stream`, `Events(filter, cursor) stream`, `EventDetail(event_id)`, `SaveQuery(name, filter)`, `ListSavedQueries()`, `DeleteSavedQuery(id)`. Server-streaming for `Stories` and `Events`; unary for the rest.

8. **Web routes:** `web/src/routes/_authed/activity.tsx` replaces the Plan 01 placeholder. Sub-components live under `web/src/pages/activity/`. The three tabs are `StoriesTab`, `AllEventsTab`, `SavedTab` — each is a separate file.

9. **Badge palette:** Interestingness category → `--sy-*` token mapping (applied via `data-interesting-category` attribute):
   - `failure` → `--sy-color-bad`
   - `performance` → `--sy-color-warn`
   - `causation` → `--sy-color-purple`
   - `anomaly` → `--sy-color-info`
   - `security` → `#d4b6e8` (lilac — no token yet; a `--sy-color-lilac` token is introduced in this plan alongside the existing palette)
   - `novelty` → `--sy-color-good`
   - `configuration` → `--sy-color-purple`

10. **Mock mode:** `ActivityService` accepts a `mock bool` config flag (set via daemon config or environment variable `SY_ACTIVITY_MOCK=1`). When true, the server returns synthetic stories generated by `internal/activity/mock/`. The web client reads the same stream — no client-side branching.

### Task 3.3 — Implement `Detector` interface + Performance detector

**Files:** `internal/interestingness/detector.go`, `performance.go`, `performance_test.go`

```go
type Detector interface {
    Category() string
    Examine(ctx context.Context, e eventstore.Event) ([]Tag, error)
}
type Tag struct{ Category, Name, Explanation string }
```

`PerformanceDetector` tracks a rolling SLO per command kind (default 2 s). Tags `slow_ack` when round-trip exceeds SLO, `timeout` for timed-out commands, `queue_depth_spike` when projection lag exceeds threshold.

**TDD:** (1) 500 ms under 2 s SLO → no tag. (2) 3200 ms → `slow_ack`. (3) Timed-out event → `timeout`.

**Acceptance:** `go test ./internal/interestingness/...` green.

**Commit:** `feat(interestingness): Detector interface + performance detector`

### Task 3.4 — Implement remaining six detectors

**Files:** `anomaly.go`, `causation.go`, `failure.go`, `security.go`, `configuration.go`, `novelty.go` and their `_test.go` counterparts.

- **Anomaly:** detect value-out-of-band (requires a per-entity min/max window from recent history) and dormant entity re-appearing (last seen > configurable interval).
- **Causation:** detect high fan-out (downstream event count > configurable threshold), automation triggers, and state changes without a preceding command within the correlation window.
- **Failure:** detect `cmd.failed`, `driver.disconnected`, `auth.failed`, `policy.denied` event kinds.
- **Security:** detect commands from unknown sources, events after configurable quiet hours, and repeated `auth.failed` events from the same source within a rolling window.
- **Configuration:** detect `config.applied`, `driver.restarted`, `automation.deployed`, `widgetpack.installed` event kinds.
- **Novelty:** maintain a "seen entities" set (persisted as projection state); tag first-seen entities, first-seen attributes, and commands last seen > 14 days ago.

Each detector must have at least two unit tests covering its primary firing condition and its non-firing condition.

**Acceptance:** `go test ./internal/interestingness/...` green; no race detector failures (`go test -race`).

**Commit:** `feat(interestingness): all six remaining detectors`

### Task 3.5 — Implement the interestingness pipeline

**File:** `internal/interestingness/pipeline.go`, `pipeline_test.go`

`Pipeline` subscribes to the event store via `eventstore.Subscribe`, fans out each incoming event to every registered `Detector` in parallel, collects non-empty tag slices, and appends one `interestingness.tagged` event per (source event, detector) pair that fired. Pipeline state is durable: it tracks its last-processed sequence number in a projection table so restarts resume without reprocessing.

**Acceptance:** integration test using `eventstoretest.NewMemStore`: emit 3 events, 2 of which trigger the `FailureDetector`; assert 2 `interestingness.tagged` events are appended. `go test -race ./internal/interestingness/...` green.

**Commit:** `feat(interestingness): detector pipeline with durable sequence tracking`

### Task 3.6 — Implement `ActivityService` (server-side)

**Files:** `internal/activity/service.go`, `stories/coalescer.go`, `stories/story.go`, `saved_queries.go`, `mock/mock.go`

**Coalescer:** given events sharing a `correlation_id`, produces a `Story` by identifying the initiating command (earliest in sequence), collecting state-change events within the window, joining `interestingness.tagged` events, and deriving a title from the command description + entity list.

**ActivityService:** `Stories` queries correlation groups in the time window, runs the coalescer, streams in reverse-chronological order. `Events` queries with filter and joins tags. `EventDetail` returns a single event with tags and causation chain. `SaveQuery` / `ListSavedQueries` / `DeleteSavedQuery` are Pkl-filesystem CRUD. When `mock=true`, `Stories` and `Events` delegate to `mock/mock.go` (all seven categories represented).

**Acceptance:** `go build ./internal/activity/...` succeeds; service registers without panic.

**Commit:** `feat(activity): ActivityService + coalescer + saved queries`
